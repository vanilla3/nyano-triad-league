# Nyano Triad League UI Guidelines v1 (JA)

このガイドは「Nyanoらしい、かわいくて、でも遊びやすい」UIのための設計原則です。
詳細な理論背景は `docs/03_frontend/UI_ANALYSIS.md` を参照してください。

---

## 1. UIの最重要原則（実装方針）

### 原則A: 視認性が最優先
かわいさは“読みやすさ”の上に乗ります。
- コントラスト
- 情報の階層
- 余白

### 原則B: プログレッシブ開示（段階的に見せる）
視聴者・プレイヤー・配信者で必要情報が違います。
- 視聴者: 要点だけ（Overlay controls=0）
- 配信者: 理由と詳細（Overlay controls=1）
- 運用者: 失敗原因と復旧導線（Stream）

### 原則C: “説明できるゲーム” に寄せる
ブロックチェーン/配信文脈では「納得できる」が価値になります。
- 何が起きたか（結果）
- なぜ起きたか（理由バッジ）
- 次に何ができるか（合法手）

---

## 2. Nyanoらしさの要素（世界観）

### 目指す雰囲気
- かわいい（だが幼すぎない）
- 清潔感
- 小さな遊び心（マイクロデライト）

### 具体的な表現案
- Nyano 画像を “空状態” や “説明” に添える
- 吹き出し（短いセリフ）
- ボタンやバッジに「丸み」「柔らかさ」を入れる

---

## 3. コンポーネント設計の作法

### Pages は “状態” / Components は “見た目”
- pages: protocol と状態の統合（難しいところ）
- components: かわいさと分かりやすさ（楽しいところ）

### 禁止に近いこと
- UI変更で protocol schema を壊す
- runtime error を残したままマージする

---

## 4. 画面ごとのUI目標（短期）

### /match
- 「今は誰のターン？」が即わかる
- 手札→盤面→結果の視線誘導
- flip/chain の短い演出

### /stream
- “今オペレーション上の問題は何？” が3秒でわかる
- allowlist件数 / hash / 投票残り時間が見える
- 失敗時の復旧導線（CORS、未接続など）

### /overlay
- 視聴者は短い要点だけ
- 配信者は詳細を見られる（controls=1）
